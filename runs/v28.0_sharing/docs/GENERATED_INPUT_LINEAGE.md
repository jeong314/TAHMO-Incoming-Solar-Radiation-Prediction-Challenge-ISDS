## v16.6 and v17.7

These are documented in more detail in:

```text
UPSTREAM_V166_V177.md
```

Short version:

- `v16.6` creates `v166_val_predictions.csv` and `v166_submission.csv`.
- `v17.7` creates `v177_val_predictions.csv` and `v177_submission.csv`.
- v28.0 uses v166/v177 validation predictions to evaluate candidate correction shapes.
- v28.0 uses v166/v177 submissions as diagnostic/reference anchors, while the actual v28.0 submitted candidate is anchored to `jieun20.csv`.

## What is required for exact v28.0 reproduction?

For exact v28.0 reproduction, users do not need to rerun v20/v16.6/v17.7.
They need the frozen CSV artifacts listed in `../data/ARTIFACT_MANIFEST.csv`.
Those artifacts are kept outside normal Git history because several are large.

For full historical reproduction from raw/project-level data, users need to run the upstream stages in lineage order:

```text
v20 physical preprocessing
v15.11 pvlib residual stack
v16.1 daily energy assimilation
v16.5/v16.6 pv residual stack
v17.3/v17.5/v17.6/v17.7 correction blend and station filter
v28.0 final edge kt correction
```

The final GitHub package should make clear which mode it supports:

- exact final-stage reproduction: supported once the manifest-listed artifacts are placed next to `v28_0_reproduce.py`;
- full historical clean reproduction: requires upstream scripts and their input artifacts.
